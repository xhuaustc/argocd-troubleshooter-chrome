# ArgoCD 排障 Chrome 插件 -- 产品设计探索报告

## 一、用户画像与使用场景

### 1.1 目标用户

| 用户角色 | 典型背景 | 使用频率 | 排障能力 | 核心诉求 |
|---------|---------|---------|---------|---------|
| **SRE / 平台工程师** | 负责 Kubernetes 集群运维，熟悉 ArgoCD 配置与管理 | 每天 | 高 | 快速定位根因，减少 MTTR |
| **DevOps 工程师** | 负责 CI/CD 流水线，维护 Helm Chart / Kustomize 配置 | 每天 | 中高 | 快速判断是流水线问题还是集群问题 |
| **应用开发者** | 通过 GitOps 部署自己的服务，对 K8s 了解有限 | 每周数次 | 中低 | 得到可操作的修复指引，减少对 SRE 的依赖 |
| **技术负责人 / Team Lead** | 需要在事故中快速了解状况 | 偶尔 | 中 | 快速获得问题摘要与影响范围 |

**核心用户**：应用开发者和初中级 DevOps 工程师。这类用户具备基本的 K8s 知识，但缺乏系统的排障经验，最能从 LLM 辅助诊断中获益。

### 1.2 典型排障场景

#### 场景 A：Sync 失败 (Sync Failed)

```
触发条件：用户点击 Sync 后状态变为 "Sync Failed"
常见原因：
- YAML 语法错误或 Schema 验证失败
- RBAC 权限不足（ServiceAccount 无法创建某些资源）
- 资源冲突（已有同名资源被其他 Application 管理）
- Helm values 渲染错误
- Kustomize overlay 合并失败
- CRD 缺失（引用了未安装的 CustomResourceDefinition）
- 命名空间不存在
```

**用户痛点**：Sync 错误信息通常是 kubectl apply 的原始输出，包含大量冗余信息，根因可能被淹没在多个资源的错误中。

#### 场景 B：健康检查异常 (Degraded / Progressing)

```
触发条件：资源 Sync 成功但健康状态非 Healthy
常见原因：
- Pod CrashLoopBackOff（镜像拉取失败、启动命令错误、OOMKilled）
- Deployment 副本数不足（PDB 限制、节点资源不足）
- Service 无可用 Endpoint
- Ingress 配置错误（后端服务不存在、TLS 证书问题）
- PVC Pending（StorageClass 不匹配、存储容量不足）
- HPA 无法获取 Metrics
```

**用户痛点**：需要逐层排查（Deployment -> ReplicaSet -> Pod -> Container -> Events），信息分散在多个资源中。

#### 场景 C：资源 OutOfSync

```
触发条件：Git 仓库定义与集群实际状态不一致
常见原因：
- 有人直接 kubectl edit 修改了资源（漂移 drift）
- Webhook / Mutating Admission Controller 修改了资源
- ArgoCD 忽略规则配置不当
- 资源被其他控制器修改（如 HPA 修改副本数）
```

**用户痛点**：OutOfSync diff 虽然显示了差异，但用户不确定是应该 Sync（覆盖集群状态）还是更新 Git 仓库。

#### 场景 D：Application 长时间 Progressing

```
触发条件：部署超过预期时间仍显示 Progressing
常见原因：
- Pod 调度失败（节点亲和性、污点容忍、资源配额）
- Init Container 阻塞
- Readiness Probe 不通过
- Finalizer 阻止资源删除
- Job 持续执行
```

**用户痛点**：Progressing 状态没有明确的错误信息，用户不知道要等多久或者是否已经出了问题。

### 1.3 用户当前排障痛点总结

1. **信息分散**：诊断一个问题需要在 ArgoCD UI 中点击多个资源、查看 Events、Logs、Diff，甚至切换到 kubectl 命令行。
2. **根因隐蔽**：表面的错误（如 Deployment Degraded）可能根因在更深层（如 Pod 的某个 Init Container 的某个 Event 描述了 ConfigMap 不存在）。
3. **知识壁垒**：不同类型的排障需要不同的 K8s 知识，而开发者通常只熟悉自己常遇到的问题类型。
4. **上下文切换**：从 ArgoCD UI 到 kubectl，再到文档查阅、日志系统搜索，打断了排障思路。
5. **缺乏可操作指引**：即使找到了原因，开发者也不确定应该修改哪个文件、哪个字段。

## 二、用户工作流程分析

### 2.1 ArgoCD Application Detail 页面结构

ArgoCD Application Detail 页面是核心工作页面，URL 模式为：
```
https://<argocd-host>/applications/<namespace>/<app-name>
```
（旧版本可能是 `/applications/<app-name>`）

页面主要组成部分：

```
+--------------------------------------------------------------------+
|  顶部栏：App名称 | Project | Sync状态 | 健康状态 | 操作按钮         |
|  操作按钮：Sync | Refresh | Delete | App Details                    |
+--------------------------------------------------------------------+
|  工具栏：视图切换(Tree/List/Pods) | 过滤器 | 搜索                   |
+--------------------------------------------------------------------+
|                                                                      |
|   资源树 / 资源列表 / Pod 视图                                        |
|                                                                      |
|   每个资源节点显示：                                                   |
|   - 资源类型图标（Deployment, Service, Pod 等）                       |
|   - 资源名称                                                         |
|   - 同步状态标记（Synced / OutOfSync）                                |
|   - 健康状态标记（Healthy / Degraded / Progressing / Missing）        |
|                                                                      |
+--------------------------------------------------------------------+
|  点击资源后的侧边面板 / 弹出面板：                                      |
|  - Summary（基本信息）                                                |
|  - Diff（期望 vs 实际）                                               |
|  - Events（K8s Events）                                              |
|  - Logs（容器日志，仅限 Pod）                                         |
|  - Manifest（完整 YAML）                                             |
+--------------------------------------------------------------------+
```

### 2.2 现有排障流程

**典型排障流程（以 Sync 失败为例）：**

```
步骤 1：进入 Application Detail 页面
  | 看到顶部 Sync 状态显示红色 "Sync Failed"
步骤 2：点击 Sync 状态或查看 Sync Result
  | 查看 Sync 操作的错误信息
步骤 3：识别哪些资源有问题
  | 在资源树中找到标红的资源
步骤 4：逐个点击异常资源
  | 查看 Events / Diff / Manifest
步骤 5：根据经验判断根因
  | 可能需要切换到命令行进一步排查
步骤 6：修改 Git 仓库中的配置
  | 提交、推送
步骤 7：重新 Sync
  | 查看是否修复
```

**现有流程的问题：**
- 步骤 3-5 高度依赖用户经验，缺乏指引
- 步骤 5 经常需要来回跳转查看多个资源
- 步骤 5-6 之间存在知识鸿沟：知道了原因，但不确定该改什么

### 2.3 插件融入工作流的方式

**目标：不改变用户现有习惯，在需要帮助时提供即时辅助**

```
改进后的流程：

步骤 1：进入 Application Detail 页面
  | 看到异常状态
步骤 2：点击插件的"排查"按钮
  | 插件自动收集资源树、Events、状态信息
步骤 3：LLM 分析并展示结果
  | 问题原因 + 修复步骤 + 定位链接
步骤 4：点击定位链接跳转到具体资源
  | 直接查看需要修改的字段
步骤 5：修改 Git 仓库中的配置
  | 提交、推送、Sync
```

**关键设计原则**：
- **非侵入式**：不修改 ArgoCD 页面的原有元素
- **按需触发**：只在用户主动点击时才收集信息和调用 LLM
- **上下文感知**：自动根据当前页面状态决定收集哪些信息
- **可操作导向**：给出的建议必须包含具体的操作路径

## 三、UI/UX 设计方案

```
+--------------------------------------------------------------------+

|                                                                      |
|                                                                      |
|                                                       [排查]         |  <-- 浮动按钮（右下角）
|                                                                      |
+--------------------------------------------------------------------+

点击后展开侧边面板：

+------------------------------------------+-------------------------+
|  ArgoCD 原有界面                          |  排障助手面板            |
|  （宽度自适应缩小或被遮挡）                 |                         |

|                                          |  -> 定位到资源B          |
|                                          |                         |

```

### 3.2 浮动按钮设计

**位置**：页面右下角，距离底部和右侧各 20px

**状态设计**：

| 按钮状态 | 视觉表现 | 说明 |
|---------|---------|------|
| 就绪 | 圆形按钮，ArgoCD 蓝色主题色 + 诊断图标 | 可点击触发排查 |
| 分析中 | 按钮带旋转动画，脉冲效果 | LLM 正在处理 |
| 有结果 | 按钮带徽标数字（问题数量） | 提醒用户查看结果 |
| 未配置 | 按钮带警告标记 | 首次使用需配置 LLM |

**可拖拽**：用户可以拖拽按钮到任意位置，位置记忆在 localStorage 中。

**自动检测**：按钮仅在 ArgoCD Application Detail 页面显示。通过 URL 匹配规则：
```
匹配模式：
  /applications/<namespace>/<app-name>
  /applications/<app-name>  (旧版)
```

### 3.5 排障结果展示方案

#### 流式输出

LLM 的回复以流式方式逐步展示，用户无需等待完整响应。面板使用 Markdown 渲染，支持：
- 标题层级
- 代码块（YAML 高亮）
- 表格
- 加粗/斜体重点

#### 结构化输出模板

通过 System Prompt 引导 LLM 按固定结构输出：

```markdown
## 问题摘要
[一句话概括问题]

## 严重程度
[Critical / Warning / Info]

## 根因分析
[详细分析，包含因果链]

## 受影响资源
- [资源类型/名称] -> [状态] -> [问题描述]

## 修复步骤
1. [步骤描述]
   - 目标资源：[资源类型/名称]
   - 当前值：xxx
   - 建议值：yyy
   - 文件位置提示：[如果知道 Git 仓库结构]

2. [步骤描述]
   ...

## 验证方法
[修复后如何验证问题已解决]
```

### 3.6 修复建议与链接的交互设计

这是插件的核心差异化功能。

#### 资源定位链接

在诊断结果中，每个提到的 K8s 资源都生成可点击的链接。点击后的行为：

```
方式 1：打开 ArgoCD 内置的资源详情面板
  -> 模拟点击 ArgoCD 资源树中对应节点
  -> 资源树滚动到该节点并高亮
  -> 自动打开该资源的侧边详情

方式 2：跳转到资源的特定 Tab
  -> 比如 "查看 Events" 直接打开 Events Tab
  -> "查看 Diff" 直接打开 Diff Tab
  -> "查看日志" 直接打开 Logs Tab
```

#### 实现策略

ArgoCD 的前端是 React 应用，资源树的节点渲染为 DOM 元素。插件可通过以下方式定位：

```javascript
// 策略 1：通过 data 属性或类名查找资源节点
// ArgoCD 资源树节点通常包含 data-resource-name 等属性
const node = document.querySelector(
  `[data-resource-name="${resourceName}"][data-resource-kind="${kind}"]`
);

// 策略 2：通过 URL Hash 导航
// ArgoCD 支持 URL 参数定位到特定资源
// 例如：?resource=<group>/<kind>/<namespace>/<name>
// 或通过 node 参数：?node=<kind>/<name>

// 策略 3：遍历资源树 DOM 文本匹配
// 在资源树中查找包含特定资源名称的节点元素
```

#### 链接样式

```
修复建议中的链接样式：

  -> 查看 Pod: api-server-85d7f9c4d-x7k2m     [详情] [Events] [日志]
     ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~   可点击标签组

  -> 定位到 Deployment: api-server 的镜像配置   [定位]
     ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~      点击后滚动并高亮
```

### 3.7 不同 ArgoCD 版本适配

| 版本范围 | UI 框架 | URL 模式 | 适配策略 |
|---------|---------|---------|---------|
| v2.0 - v2.5 | React (旧版) | `/applications/<app-name>` | 通过 DOM 特征检测 |
| v2.6 - v2.9 | React (新版) | `/applications/<namespace>/<app-name>` | 主要适配目标 |
| v2.10+ | React (最新) | `/applications/<namespace>/<app-name>` | 主要适配目标 |

**适配方法**：
1. 通过 ArgoCD 页面 meta 标签或 API 端点 `/api/version` 获取版本号
2. 维护不同版本的 DOM 选择器映射表
3. 对于资源定位功能，优先使用 URL 参数方式（跨版本更稳定），降级到 DOM 查找

## 四、信息收集策略

### 4.1 收集信息清单

按优先级从高到低排列：

#### P0 - 必须收集（基本诊断所需）

| 信息类型 | 来源 | 收集方式 | 说明 |
|---------|------|---------|------|
| Application 状态 | 页面 DOM / API | DOM 解析或 API `/api/v1/applications/{name}` | Sync 状态、健康状态、操作结果 |
| 资源树 | 页面 DOM / API | API `/api/v1/applications/{name}/resource-tree` | 所有管理的资源及其状态 |
| Sync 错误信息 | 页面 DOM | 解析 Sync Result 区域 | Sync 失败的具体错误 |
| 资源健康详情 | API | 从资源树数据提取 | 每个资源的健康状态和消息 |

#### P1 - 推荐收集（深度诊断所需）

| 信息类型 | 来源 | 收集方式 | 说明 |
|---------|------|---------|------|
| K8s Events | API | API `/api/v1/applications/{name}/events` 或资源级别 Events | Warning 级别事件是重要诊断线索 |
| 资源 Diff | 页面 DOM / API | API `/api/v1/applications/{name}/managed-resources` with diff | OutOfSync 资源的差异内容 |
| Conditions | API | Application Status Conditions | 如 SyncError, InvalidSpecError 等 |
| Operation State | API | Application Operation State | 最近一次 Sync/Rollback 操作的详细结果 |

#### P2 - 可选收集（需用户授权）

| 信息类型 | 来源 | 收集方式 | 说明 |
|---------|------|---------|------|
| 容器日志 | API | API `/api/v1/applications/{name}/logs` | 最近 N 行日志，可能含敏感数据 |
| 完整 Manifest | API | 资源的 Live Manifest | 可能含 Secret 引用 |
| Application 参数 | API | Helm values / Kustomize params | 可能含配置密钥 |

### 4.2 信息收集流程

```
用户点击"开始排查"
       |
  +-- 阶段 1：快速收集（< 2秒）
  |  |-- 解析页面 DOM 获取 Application 名称、命名空间
  |  |-- 获取页面显示的 Sync 状态、健康状态
  |  +-- 获取页面上可见的错误信息
  |
  +-- 阶段 2：API 深度收集（< 5秒）
  |  |-- 调用 Application Detail API 获取完整状态
  |  |-- 调用 Resource Tree API 获取资源树
  |  |-- 对异常资源获取 Events
  |  +-- 对 OutOfSync 资源获取 Diff
  |
  +-- 阶段 3：按需收集（用户授权后）
  |  |-- 获取 CrashLoopBackOff Pod 的日志
  |  +-- 获取特定资源的完整 Manifest
  |
  +-- 阶段 4：组装 Prompt，调用 LLM
```

### 4.3 API 访问认证

Chrome 插件运行在 ArgoCD 页面的上下文中，可以复用浏览器当前的认证会话：

```
方式 1：Cookie 认证
  -> ArgoCD UI 登录后浏览器有 argocd.token Cookie
  -> 插件的 fetch 请求自动携带 Cookie（同源）

方式 2：Bearer Token
  -> 从页面 localStorage 中获取 token
  -> ArgoCD 前端通常将 token 存在 localStorage
  -> 在请求头中添加 Authorization: Bearer <token>
```

### 4.4 敏感信息处理

**脱敏策略**：

```
1. Secret 类资源
   -> 只收集 metadata（名称、类型），不收集 data
   -> 在 Prompt 中标注 "Secret 内容已脱敏"

2. ConfigMap 中可能含密码的字段
   -> 正则匹配 key 名（password, secret, token, key, credential）
   -> 替换 value 为 "[REDACTED]"

3. 容器日志
   -> 用户明确授权后才收集
   -> 提示用户日志将发送给 LLM 服务

4. 环境变量
   -> 只收集变量名，不收集值
   -> 标注来源（ConfigMap / Secret / 直接定义）
```

## 五、竞品与替代方案分析

### 5.1 市场现状

目前市场上**没有直接针对 ArgoCD 的 LLM 排障 Chrome 插件**。相关的工具和方案包括：

#### 同类产品

| 产品/工具 | 类型 | 覆盖范围 | 与本插件的差异 |
|-----------|------|---------|--------------|
| **K8sGPT** | CLI 工具 / Operator | 通用 K8s 排障 | 命令行工具，不与 ArgoCD UI 集成；需要集群访问权限；不感知 GitOps 上下文 |
| **Komodor** | SaaS 平台 | K8s 问题诊断 + 变更追踪 | 完整平台，需要安装 Agent；功能强大但部署和费用成本高 |
| **Robusta** | K8s 可观测平台 | 事件驱动的自动化排障 | 需要集群内安装；侧重自动化，而非交互式辅助 |
| **Kubecost + ChatGPT** | 组合使用 | 成本 + 排障 | 需要手动复制信息到 ChatGPT |
| **kubectl-ai** | kubectl 插件 | K8s 命令辅助 | 命令行工具，帮助生成 kubectl 命令 |

#### 替代工作流

| 方式 | 步骤 | 优势 | 劣势 |
|------|------|------|------|
| **手动 + 文档查阅** | 查看状态 -> 搜索文档 -> 逐步排查 | 无额外工具依赖 | 耗时长，依赖经验 |
| **手动 + ChatGPT** | 复制错误信息 -> 粘贴到 ChatGPT | 灵活性高 | 上下文切换多，需要手动收集信息，信息可能不完整 |
| **Runbook / Playbook** | 按预设脚本排查 | 标准化流程 | 无法覆盖所有场景，维护成本高 |
| **找 SRE 同事帮忙** | 提交工单或直接联系 | 人类专家判断 | 响应延迟，SRE 时间宝贵 |

### 5.2 本插件的差异化优势

1. **零安装成本**：Chrome 插件，无需集群内部署任何组件，无需 kubectl 权限。
2. **上下文感知**：直接在 ArgoCD 页面工作，自动收集相关信息，无需手动复制粘贴。
3. **GitOps 语义**：理解 ArgoCD 的 Sync/Health 语义，比通用 K8s 工具更精确。
4. **可操作链接**：修复建议直接链接到 ArgoCD UI 中的资源，缩短从"知道问题"到"开始修复"的路径。
5. **模型灵活**：用户自选 LLM 服务，可使用企业内部私有模型，解决数据安全顾虑。
6. **轻量级**：不需要持久化存储、不需要集群权限、不收集遥测数据。

### 5.3 潜在风险与挑战

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| ArgoCD UI 变更导致 DOM 解析失败 | 核心功能受损 | 优先使用 API 而非 DOM；维护多版本适配 |
| LLM 给出错误建议 | 用户误操作 | 强调"建议仅供参考"；展示原始收集信息供用户验证 |
| API 认证过期 | 信息收集失败 | 优雅降级，提示用户刷新登录 |
| 信息不足导致诊断不准 | 用户体验差 | 允许用户追加提问，渐进式收集更多信息 |
| 企业安全审查不通过 | 无法在企业内使用 | 支持私有 LLM、数据脱敏、开源审计 |

## 六、设计优先级与 MVP 范围

### 6.1 MVP（最小可用产品）

**目标**：验证核心价值假设 -- "自动收集上下文 + LLM 诊断" 是否对用户有帮助。

MVP 功能范围：

1. **页面检测**：仅在 ArgoCD Application Detail 页面激活
2. **浮动按钮**：固定位置，点击展开侧边面板
3. **LLM 配置**：支持 OpenAI 兼容 API 的基本配置（endpoint, key, model）
4. **信息收集**：通过 ArgoCD API 收集 Application 状态、资源树、Events
5. **诊断展示**：流式输出 Markdown 格式的诊断结果
6. **资源定位**：诊断结果中的资源名称可点击，跳转到 ArgoCD 资源详情

**不在 MVP 中的功能**：
- 历史记录
- 数据脱敏
- 多语言支持
- 追问/对话功能
- 容器日志收集
- 反馈系统

### 6.2 后续迭代方向

- **V1.1**：数据脱敏、高级 LLM 参数配置、用户反馈机制
- **V1.2**：追问/对话功能、历史记录、容器日志收集
- **V1.3**：多 Application 对比诊断、团队共享配置
- **V2.0**：预设 Prompt 模板市场、自定义排障规则

## 七、设计总结

本插件的核心产品逻辑可以概括为：

```
减少排障路径 = 自动收集上下文 + LLM 智能分析 + 可操作修复链接
```

**设计理念**：
- 不替代用户思考，而是加速信息收集和初步分析
- 不改变现有工作流，而是在关键节点提供辅助
- 不锁定特定 LLM 服务，而是给用户选择权
- 不追求大而全，而是在 ArgoCD 排障这个垂直场景做到极致

**成功指标**：
- 排障平均时间缩短 50% 以上
- 开发者对 SRE 的排障求助减少 30%
- 用户 "有帮助" 反馈率 > 70%
